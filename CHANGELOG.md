# Changelog

## perfetto-mcp-rs 0.x Changes

### [0.13.4](https://github.com/tooluse-labs/perfetto-mcp-rs/releases/tag/v0.13.4) (May 6, 2026)

- **Silent output for clients that aren't installed.** v0.13.3 and earlier emitted a `==> Codex skipped: codex not installed` line for every probed client whose CLI / app / config dir was absent. Users who don't run a given client (e.g. Codex on a Claude-only setup) saw this as noise — telling them about a tool they don't use, every install / uninstall. New `Outcome::Absent` variant takes over the "client not installed at all" cases (`register_claude` / `register_codex` / `register_qoder` and their deregister counterparts) and produces zero output. The wrapper still proceeds normally (exit 0); the user just sees clean output for the clients that ARE installed.
- **`Outcome::Skipped` semantics tightened**: now reserved for "user explicitly opted out" cases (`--skip-claude`, `--skip-codex`, `--skip-qoder`, `--keep-cache`) and benign uninstall conditions (`no user-scoped registration to remove`, `cache root unavailable`). Those still echo so users see their flags acknowledged. `Outcome::Manual` (client present but needs paste action) is unchanged. `Outcome::Failed` (real error) still prints to stderr.
- **Test contract update**: `install_cli_missing_skips_gracefully` now asserts the `==> Claude` / `==> Codex` lines do NOT appear when those CLIs aren't on PATH (the inverse of the old test, which asserted they did appear). New unit test `aggregate_treats_absent_as_silent_success` pins the silent-on-Absent + exit-0 contract so a future regression flipping Absent back to a printing variant is caught at compile time.
- **No code-path changes** beyond the output filtering. Detection logic, install/uninstall flow, env var passthrough, and Manual-outcome semantics carry forward from v0.13.3 unchanged.

### [0.13.3](https://github.com/tooluse-labs/perfetto-mcp-rs/releases/tag/v0.13.3) (May 6, 2026)

- **`install.sh` no longer hits the GitHub anonymous API on the latest path.** v0.13.2 and earlier resolved `VERSION=latest` through `api.github.com/repos/.../releases/latest`, then aborted with `could not resolve release tag` on rate-limit (60 req/hour per IP) or transient network failure — even though the API call wasn't strictly required to download the asset. install.ps1 has used the `releases/latest/download/<asset>` redirect (no API call) since day one, with a comment specifically calling this out. install.sh now mirrors that behavior: download URL goes through the redirect; the API call survives only as a best-effort version-display lookup with `|| true`, so a rate-limit/network blip degrades the install message from `==> Installing perfetto-mcp-rs v0.13.3 (darwin-arm64)` to `==> Installing perfetto-mcp-rs latest (darwin-arm64)` instead of aborting. Pinned-version path unchanged.
- **`claude mcp add` invocation now uses the `--` separator before the binary path.** `claude mcp add --help` documents `--` as the form for stdio-server-with-args registrations ("Add stdio server with subprocess flags: `claude mcp add my-server -- my-command --some-flag arg1`"). Our own paste-ready hint in `claude_manual_install_body` already emits `--`, and the codex `mcp add` invocation already does too — the actual claude invocation was the lone holdout. Pure inconsistency fix; aligns three previously divergent call sites. Practical risk was theoretical (binary paths land at `~/.local/bin/...`, not paths starting with `-`), but adopting the documented form prevents the wpa-mcp-style "argparse ate my path" failure mode if a future user puts the binary somewhere with a leading `-`.
- **Two integration tests** (`tests/install_subcommand.rs:178, 204`) updated to assert on the new `claude|mcp add perfetto-mcp-rs --scope user -- {bin}` recorded-call format. clippy + fmt clean. CI matrix runs unchanged.
- **No code-path changes** beyond the install/uninstall fix-ups above. Everything else from v0.13.2 (multi-client install fallbacks, blocking uninstall semantics, SKIP env-var passthrough, brand assets) carries forward unchanged.

### [0.13.2](https://github.com/tooluse-labs/perfetto-mcp-rs/releases/tag/v0.13.2) (May 6, 2026)

- **Multi-client install/uninstall fallbacks for MCP clients without first-class CLI control.** `perfetto-mcp-rs install` previously only wired up Claude Code (`claude mcp add`) and Codex (`codex mcp add`); if either CLI wasn't on PATH it silently skipped, leaving users with a freshly-downloaded binary that no client knew about. v0.13.2 adds a paste-ready-snippet fallback path for three additional surfaces, gated on positive install signals so users without a given client see no noise:
  - **Qoder** — detects via `which qoder`, `/Applications/Qoder.app` (macOS), and `%LOCALAPPDATA%\Programs\Qoder` (Windows); emits a `{"mcpServers": {"perfetto-mcp-rs": ...}}` JSON snippet for the user to paste into Qoder Settings → MCP → + Add. Qoder's MCP config is UI-only with no documented disk path, so we never claim to manage it.
  - **Codex desktop app / `~/.codex/` config dir** — when the CLI is hidden (the `curl|sh` subshell case where login-shell-only PATH additions don't propagate, or when only the Mac app is installed), emits a TOML snippet for `~/.codex/config.toml`. All Codex surfaces (CLI, VS Code ext, desktop app) read this same file.
  - **Claude Desktop / Claude Code path-hidden** — detects Claude Desktop via `/Applications/Claude.app` and the per-OS data dir (`dirs::config_dir().join("Claude")`); detects Claude Code via `~/.claude.json` / `~/.claude/`. Emits a Manual outcome with one or both sections depending on which products are present: a JSON snippet for `claude_desktop_config.json` and/or a paste-ready `claude mcp add ... --scope <user-requested-scope>` line. Desktop fallback is gated to `--scope user` because Local/Project scopes are Claude Code-only concepts without a Desktop analog.
- **Uninstall correctness: blocking Manual outcomes prevent dangling MCP entries.** When the binary's `uninstall` subcommand can't programmatically remove an entry but the client *does* have a documented config path (Codex, Claude Desktop), the new `Outcome::Manual { blocking: true }` variant exits non-zero so `uninstall.sh:70` and `uninstall.ps1` keep the binary in place. Without this, the wrapper would happily delete the binary while the client config still pointed at the deleted path — silently breaking MCP integration on the next client launch. Qoder uninstall stays `blocking: false` (informational-only) because its config path is undocumented; blocking would just trap the user with no concrete file to edit.
- **`SKIP_CLAUDE` / `SKIP_CODEX` env-var passthrough on the uninstall wrappers.** After manually cleaning up a config entry (per the printed Manual hint), users finish uninstall with `SKIP_CLAUDE=1 curl ... | sh` (or `$env:SKIP_CLAUDE='1'; irm ... | iex`). Pattern parallels the existing `SCOPE` env. No `SKIP_QODER` passthrough — Qoder uninstall doesn't block, so it isn't needed.
- **Quick install README reorg.** The `curl ... | sh` / `irm ... | iex` one-line script is now the recommended path on every platform, since it's the only path that auto-registers with Claude Code and Codex. Homebrew and `cargo install` move under "alternatives" with their manual `perfetto-mcp-rs install --binary-path` follow-up step explicitly shown. zh-CN README also gets a language polish pass — phrasings like "顺手自动注册" and "沿着错误消息的提示串起" read like machine translation; rewritten to flow as native Chinese without changing technical content.
- **`list_tables` description: SQLite engine fact anchor.** Real-world v0.13.1 session showed an LLM trying `SELECT name FROM perfetto_tables WHERE ...`, assuming `trace_processor` exposes a Postgres/BigQuery-style catalog. It doesn't — the engine is SQLite, schema introspection runs through this MCP tool (or `sqlite_master`). The "Use when:" section now anchors the underlying-engine fact so the LLM has ground truth before reaching for catalog SQL.
- **Brand refresh.** Lighter wordmark, refined trace/MCP mark for better small-size legibility, refreshed social preview (PNG + SVG resynced at 1280×640).
- **Tests**: 2 new aggregate-policy tests pin `Outcome::Manual { blocking }` semantics; 4 new claude-manual-body tests cover Desktop-only, CLI-only, both-detected, and scope-propagation paths; 2 new Qoder-outcome tests hard-assert non-blocking by construction (avoiding the vacuous-pass trap of going through `deregister_qoder` on machines without Qoder); JSON/TOML snippet builders pinned for Windows backslash escaping. clippy + fmt clean. CI matrix runs on 3 OSes; local Windows AppLocker still blocks new test-binary execs but unit tests verified in CI.
- **No breaking changes** to MCP tool schemas, install/uninstall CLI surface (existing `--scope` / `--skip-claude` / `--skip-codex` continue to work; new `--skip-qoder` is additive), or wrapper interface (existing `SCOPE` / `INSTALL_DIR` env vars unchanged; new `SKIP_*` env vars are additive). Existing v0.13.1 install + load + query flows continue without any caller-side changes.

Updated README + zh-CN main pitch and Tools table; `glama.yaml` server description and `load_trace` tool description; `src/server.rs` `LoadTraceParams` field doc, `STDLIB_INSTRUCTIONS` const, and `load_trace` `#[tool(...)]` description (visible to LLMs at `tools/list`). ### [0.12.2](https://github.com/0xZOne/perfetto-mcp-rs/releases/tag/v0.12.2) (April 30, 2026)

### [0.11.3](https://github.com/0xZOne/perfetto-mcp-rs/releases/tag/v0.11.3) (April 30, 2026)

- **`path` parameter removed from every tool except `load_trace`.** A v0.11.2 Claude Code session showed a stark failure mode: the LLM forgot the `path` field on its first `execute_sql`, then tried `chrome_main_thread_hotspots(filter: "{...}")` (hallucinated wrapper), then read `src/server.rs` twice to figure out the schema, then gave up entirely. Root diagnosis: `path` is a *routing key* in the server's mind (LRU caches multiple `trace_processor_shell` processes by path) but is *redundant context* in the LLM's mind (the trace is already loaded, why repeat the path?). v0.11.3 reflects the LLM's mental model — `load_trace` sets a server-side `current_trace`; every other tool reads it. The `tools/list` schema for those tools simply has no `path` field, so the LLM can't get it wrong. Switching between cached traces is now done via re-calling `load_trace`, which is near-zero-cost on a manager hit.
- **String-or-number coercion on numeric tool params.** The same v0.11.2 session passed `chrome_main_thread_hotspots(pid: "12800", min_dur_ms: "50", limit: "30")` — every numeric value as a JSON string. serde correctly rejected each call (type mismatch), so the LLM bounced 4 times before bailing. v0.11.3 adds `lenient_i64` / `lenient_f64` / `lenient_u32` deserializers via `#[serde(deserialize_with = ...)]` on five numeric fields (`pid`, `upid`, `min_dur_ms`, `limit` on `ChromeMainThreadHotspotsParams`; `upid` on `ListThreadsInProcessParams`). They accept JSON numbers and JSON strings holding the same value; reject everything else with an actionable message. JsonSchema is unaffected — `tools/list` still advertises `integer`/`number`, so well-behaved LLMs see strict types and the lenient path is only a server-side safety net (Postel: be conservative in what you advertise, liberal in what you accept).
- **`name` accepted as alias for `table_name` on `list_table_structure`.** Same kind of v0.10.1-style designer-side gap as the v0.11.2 `upid` symmetry fix: the LLM occasionally writes `{name: "chrome_page_loads"}` instead of `{table_name: "chrome_page_loads"}` because "name" is the more generic mental label. `#[serde(alias = "name")]` on the field accepts both with no behavior change otherwise.
- **`load_trace` is now the only entry point with a `path` parameter.** Its description gained one sentence stating "Sets the current trace — every other tool's `path` parameter then defaults to this value." The legacy `#[serde(alias = "trace_path")]` on `LoadTraceParams::path` remains for v0.10.x backward compat. Other tools' `path` fields (whether canonical or via `trace_path` alias) now correctly fail with `deny_unknown_fields` so v0.10.x callers get a clear migration error rather than silent path divergence.

### [0.8.5](https://github.com/0xZOne/perfetto-mcp-rs/releases/tag/v0.8.5) (April 28, 2026)

- **Diagnostics fix**: `load_trace` now surfaces actionable error signals when `trace_processor_shell` fails to start, instead of bottoming out at the generic 20s readiness timeout. Three independent leaks plugged in `src/tp_manager.rs`:
  - **stdout was discarded** (`Stdio::null()` at `spawn`). Some Perfetto failure paths print to stdout; `Stdio::piped()` plus a new `spawn_stdout_drain` (folds into the same shared tail with a `[stdout] ` prefix) preserves them. The `format_stderr_tail` label is updated to `output tail (stderr + stdout, last N lines)` so the LLM sees both streams are folded in.
  - **stderr drain died on the first non-UTF-8 byte**. Tokio's `BufReader::lines()` returns `InvalidData` on any byte sequence that isn't valid UTF-8 and our drain `break`'d on Err — so on a Chinese-locale Windows shell, where trace_processor_shell prints error text via the active ANSI codepage (GBK / Shift-JIS / etc.), the very first error line killed the drain and every subsequent stderr byte was lost. Replaced with a manual `read_until(b'\n')` plus `String::from_utf8_lossy`: invalid sequences become `U+FFFD` (`�`) but the drain stays alive, ASCII portions (errno codes, port numbers, file sizes, paths under cp1252-overlap) survive intact, and startup needle matching keeps working because the needles are pure ASCII.
  - **Non-ASCII trace paths failed silently as a 20s timeout on Windows**. `trace_processor_shell.exe` consumes argv via `int main(int, char*[])` — bytes are decoded against the active ANSI codepage, so any character outside that codepage (Chinese paths under non-CJK locales, Japanese names under English locales, etc.) becomes `?`, fopen fails on a path that does not exist, and the shell exits with no actionable banner. New `ensure_trace_path_supported` runs after `canonicalize` and bails before spawn with a directly-actionable error: it reproduces the offending path and points at the two known workarounds — copy/move to an ASCII-only path, or pass the 8.3 short name from `cmd /c dir /x "<parent>"`. Unix is a no-op (argv is byte-exact).
- 5 new unit tests in `src/tp_manager.rs`: lossy decode preserves ASCII prefix and replaces invalid bytes with `U+FFFD`, LF / CRLF / bare-CR handling, `format_stderr_tail` label coverage, ASCII path always accepted, Unix non-ASCII no-op, and `#[cfg(windows)]` non-ASCII rejection (runs only in the Windows CI matrix).
- **Why this still warrants the v0.8.4 20s timeout**: the two changes target disjoint failure modes. v0.8.4 raised the deadline so a healthy-but-slow startup (low-end Windows + DirectWrite cold-cache font enumeration during stdlib init) doesn't time out; v0.8.5 makes shell-crash failures fail fast with rich diagnostics. The poll loop's `try_wait()` already detects an exited child within one `READY_POLL_INTERVAL`, so crash-mode failures resolve in ~100–300ms — they never approach the 20s ceiling.

### [0.8.4](https://github.com/0xZOne/perfetto-mcp-rs/releases/tag/v0.8.4) (April 27, 2026)

- **UX fix**: `--startup-timeout-ms` default raised from `5000` to `20000` (`src/tp_manager.rs:35`, `src/main.rs:90`). A tiered retry inside the same poll loop is observationally equivalent to extending the deadline. Override entrypoints (`PERFETTO_STARTUP_TIMEOUT_MS` env, `--startup-timeout-ms` flag) unchanged from v0.8.0; CI/scripted contexts that want fast-fail can still pin a smaller value. README tables (en + zh) updated to match.

### [0.8.0](https://github.com/0xZOne/perfetto-mcp-rs/releases/tag/v0.8.0) (April 19, 2026)

- **New `install` / `uninstall` subcommands**. The binary now self-registers with Claude Code (`claude mcp add`) and Codex (`codex mcp add`) and self-cleans its `dirs::data_local_dir()` cache, replacing the register/deregister blocks that lived in `install.sh` / `install.ps1` / `uninstall.sh` / `uninstall.ps1` (four copies of the same CLI-schema + cache-path knowledge that had already drifted into bugs — e.g. the Git Bash `LOCALAPPDATA` cygpath issue fixed in v0.7.0's uninstall.sh). Default subcommand-less invocation still runs the MCP server, so existing client configs don't need to change. New flags: `install --scope user|local|project`, `install --binary-path <P>`, `uninstall --scope X [--keep-cache]`, and `--skip-claude` / `--skip-codex` on both.
- **Shell scripts shrink to distribution glue** — `install.sh` and `uninstall.sh` (plus PowerShell counterparts) now do download / platform detect / file-lock aside / PATH hint / binary delete, and delegate all Claude/Codex registration + cache cleanup to `perfetto-mcp-rs install|uninstall`. Uninstall scripts **preserve the binary on subcommand failure** so users can fix the underlying issue (locked cache, wrong project dir, broken config) and re-run — only a successful uninstall (or confirmed v0.7 "no subcommand" case, or already-missing binary) deletes the binary.
- **`SCOPE` env var** on all four scripts (default `user`) plumbs through to `--scope`. For local/project installs, run the script from the target project directory: `SCOPE=local bash install.sh` / `SCOPE=local bash uninstall.sh`. Claude stores local/project scope in a CWD-keyed way inside `~/.claude.json` / project's `.mcp.json`, so the CWD must match at both install and uninstall time.
- **No more `claude mcp list` probe on the install/uninstall paths**. We used to list before add/remove to distinguish first-install from upgrade, but `claude mcp list` is **not a passive probe**: it skips the workspace-trust dialog and spawns every visible stdio server to run a health check. Running that from `--scope local|project` in a hostile or broken project directory is unsafe (arbitrary command execution from that repo's `.mcp.json`) and unreliable (hangs / fails on any broken sibling server). v0.8.0 always calls `mcp remove` first and classifies its stderr: `No <scope>-scoped MCP server found with name: ...` (exit 1) is benign; anything else — including output that mixes corruption-recovery text with the benign line — is a hard failure. See `src/install.rs::claude_remove_error_is_not_found` + its unit tests for the exact pattern.
- **`PERFETTO_STARTUP_TIMEOUT_MS` / `PERFETTO_QUERY_TIMEOUT_MS` are read lazily inside `run_server`**, not by clap at top-level parse. A stale or invalid value in the user's shell no longer blocks `perfetto-mcp-rs install` / `uninstall` with a parse error — those maintenance commands don't care about server timeouts. Precedence preserved: explicit `--flag` > env var > built-in default; invalid env in server mode still fails loudly.
- `install.rs::run_install` absolutizes `--binary-path` via `std::path::absolute` before passing it to the MCP clients. MCP clients spawn the registered command from their own working directory, so a relative `./perfetto-mcp-rs` would silently break. We deliberately don't `canonicalize` (that would resolve symlinks and diverge from the `$INSTALL_DIR` path users see). Windows POSIX paths (`/c/Users/...` from Git Bash) are converted by the shell wrapper via `cygpath -m` before reaching the Rust binary.
- **Migration 0.7.x → 0.8.0**: re-run `install.sh` (or `install.ps1`) — the new binary brings the subcommands and subsequent `uninstall.sh` picks them up. Uninstalling a 0.7.x install with the new `uninstall.sh` only deletes the binary and prints a manual-cleanup hint, because the old binary has no `uninstall` subcommand. Manual cleanup command bundle for reference:
  ```sh
  claude mcp remove perfetto-mcp-rs --scope user
  codex  mcp remove perfetto-mcp-rs
  # Cache (pick the one for your OS):
  rm -rf "${XDG_DATA_HOME:-$HOME/.local/share}/perfetto-mcp-rs"   # Linux
  rm -rf "$HOME/Library/Application Support/perfetto-mcp-rs"      # macOS
  # Windows Git Bash/MSYS ($LOCALAPPDATA is a native 'C:\...' path that
  # POSIX rm can't walk; convert with cygpath, or use PowerShell):
  rm -rf "$(cygpath -u "$LOCALAPPDATA")/perfetto-mcp-rs"          # Git Bash
  # PowerShell: Remove-Item -Recurse -Force "$env:LOCALAPPDATA\perfetto-mcp-rs"
  ```
- Ship includes 23 hermetic integration contracts (`tests/install_subcommand.rs`) exercising the subcommands via stateful shell fixtures (`tests/fixtures/{claude,codex}`), plus 12 unit tests in `src/install.rs`. Integration tests run under a restricted `PATH=<fixtures>` so `which::which` can never find real CLIs. Design in `docs/plans/v0.8-binary-self-install.zh-CN.md`.
- `cargo install perfetto-mcp-rs` remains a **developer-fallback channel** only — it requires a system `protoc` because `build.rs` invokes `prost-build`. End users should use `install.sh` / `install.ps1`.

- [25b72b0](https://github.com/0xZOne/perfetto-mcp-rs/commit/25b72b0) Classify query errors at the decode boundary. `QueryErrorKind` is now `MissingTable` / `MissingModule` / `Other` (non-exhaustive), and the server-layer hint formatters exhaust the enum via `match` instead of fragile `msg.contains` string matching, so hint behavior no longer silently drifts when upstream error text changes.
- [8d01dba](https://github.com/0xZOne/perfetto-mcp-rs/commit/8d01dba) Milestone 2 regression coverage: tests for LRU eviction, subprocess recovery after abnormal exit, concurrent same-trace and cross-trace access, and the server hint paths for missing-table / missing-module failures.
- [8eaba62](https://github.com/0xZOne/perfetto-mcp-rs/commit/8eaba62) End-to-end smoke test that drives `spawn → HTTP RPC → decode` against a real trace fixture on every CI run, replacing the prior "single-chain-works" assumption with an automated check.
- [7aeaae6](https://github.com/0xZOne/perfetto-mcp-rs/commit/7aeaae6) Milestone 1 runtime hardening: instance lifecycle, spawn / wait-ready coordination, port allocation, and trace-identity checks on `/status` are reworked for correctness under concurrent MCP traffic.
- [a7c648e](https://github.com/0xZOne/perfetto-mcp-rs/commit/a7c648e) New domain analysis tools built on top of the generic `execute_sql` executor: process and thread summaries, Chrome scroll-jank summary, and a stdlib URL hint for module discovery.
- [68a3954](https://github.com/0xZOne/perfetto-mcp-rs/commit/68a3954) Borrow curated LLM nudges from the `com.google.PerfettoMcp` plugin for missing-table / missing-module errors, giving agents actionable hints instead of raw `trace_processor` error strings.
- [8bf9197](https://github.com/0xZOne/perfetto-mcp-rs/commit/8bf9197) Filter internal (`__`-prefixed) tables from `list_tables` output, and nudge aggregation when `execute_sql` would otherwise overflow the row cap.
- [7c609e9](https://github.com/0xZOne/perfetto-mcp-rs/commit/7c609e9) Native PowerShell installer `install.ps1` for Windows, complementing the existing POSIX `install.sh`. Supports ConstrainedLanguage mode, survives first-install `claude mcp remove` non-zero exit, and renames locked `.exe` aside to allow in-place upgrades.

### [0.1.1](https://github.com/0xZOne/perfetto-mcp-rs/releases/tag/v0.1.1) (April 12, 2026)

- [7552f18](https://github.com/0xZOne/perfetto-mcp-rs/commit/7552f18) Defer the `trace_processor_shell` download until the first MCP tool call instead of eagerly downloading at startup, so `claude mcp add` completes even on restricted networks. On Windows, append `.exe` to the cached binary name so the cache path matches the installed file.
- [7f54864](https://github.com/0xZOne/perfetto-mcp-rs/commit/7f54864) `install.sh` automatically adds the install directory to the Windows user `PATH`, with forward-slash paths so Git Bash / MSYS2 / Cygwin can source it without further quoting.
- [a3b9c4e](https://github.com/0xZOne/perfetto-mcp-rs/commit/a3b9c4e) `install.sh` supports Windows via Git Bash, MSYS2, and Cygwin: binary-name detection, platform-arch mapping, and path normalization all branch correctly under the three POSIX-on-Windows shells.

### [0.1.0](https://github.com/0xZOne/perfetto-mcp-rs/releases/tag/v0.1.0) (April 12, 2026)

- [e61bd86](https://github.com/0xZOne/perfetto-mcp-rs/commit/e61bd86) Initial public release of `perfetto-mcp-rs`: an MCP server for Perfetto trace analysis, backed by an auto-downloaded `trace_processor_shell`. Exposes four core tools — `load_trace`, `execute_sql`, `list_tables`, `table_structure` — over the MCP stdio transport.
- [8fbceb7](https://github.com/0xZOne/perfetto-mcp-rs/commit/8fbceb7) Process lifecycle management: spawn `trace_processor_shell` per trace, wait for readiness on stderr, reclaim ports and processes through an LRU instance pool, and re-spawn automatically after abnormal child exit.
- [7a69ce0](https://github.com/0xZOne/perfetto-mcp-rs/commit/7a69ce0) HTTP RPC client plus protobuf query-result decoder that talks to the `trace_processor_shell` loopback server and converts its binary response into `serde_json::Value` rows for MCP callers.
- [a2e9441](https://github.com/0xZOne/perfetto-mcp-rs/commit/a2e9441) Fix the MCP handshake against `claude mcp add` and add the initial CI / release scaffold so cross-platform builds and release assets can be produced from `main`.
