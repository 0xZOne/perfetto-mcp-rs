# Changelog

## perfetto-mcp-rs 0.x Changes

### [0.10.0](https://github.com/0xZOne/perfetto-mcp-rs/releases/tag/v0.10.0) (April 29, 2026)

- **Reverts v0.9.0's `rmcp::Json<T>` return-type migration on the 10 data-returning MCP tools.** Tabular tools (`execute_sql`, `list_processes`, `list_threads_in_process`, the 5 `chrome_*`) and the schema-discovery tools (`list_tables`, `list_table_structure`) all return `Result<String, String>` again, with the body produced via `serde_json::to_string(&value)` against the same typed `DecodedTable` / `TableList` / `TableInfo` / `ColumnInfo` structs introduced in v0.9.0. Wire shape unchanged at the byte level — still compact `{columns:[...],rows:[[v,...],...]}` for tabular tools, `{names:[...]}` for `list_tables`, `{table,columns:[{name,type,nullable}]}` for `list_table_structure`.
- **Why**: `rmcp::Json<T>` filled `CallToolResult.structured_content` with the parsed object alongside `content[0].text`, and Claude Code's UI renders `structured_content` as multi-line pretty-printed JSON — every cell on its own indented line. A 30-row × 8-column result blew up to ~340 rendered lines, and a single agent analysis chain (16 tool calls) buried the conversation under thousands of pretty-printed lines. Codex renders the same wire response compactly because it falls back to `content[0].text` rather than expanding `structured_content`. The `Json<T>` path was added in v0.9.0 to register `outputSchema` for free; the UX cost on the dominant client (Claude Code) outweighs the formal-contract benefit.
- **`outputSchema` removed from `tools/list`** as a side effect — there is no longer a typed `Json<T>` return for rmcp's macro to auto-derive a schema from. Agents that consumed `outputSchema` programmatically (Claude API tool-use callers) infer the response shape from the first tool result, which is self-describing JSON. New `no_tool_carries_output_schema` regression test pins the absence so a future re-introduction of `Json<T>` becomes a deliberate, visible change.
- **All other v0.9.0 work preserved**: decoder-direct columnar output, the typed `DecodedTable`, the `pragma_row_to_column_info` helper that surfaces SQLite contract violations as `Err`, column-ordering-from-`SELECT`-clause behavior, every other regression test. The change is mechanical at the rmcp adapter layer; everything below `tp_client::query` is untouched.
- 119 tests pass (120 from v0.9.0 + 1 new `no_tool_carries_output_schema` − 2 obsolete `output_schema_*` tests). `cargo clippy --lib --all-targets -D warnings` clean; `cargo fmt --all --check` clean.

### [0.9.0](https://github.com/0xZOne/perfetto-mcp-rs/releases/tag/v0.9.0) (April 28, 2026)

- **Decoder rewrite eliminates the row-of-objects intermediate.** `decode_query_result` now returns a typed `DecodedTable { columns: Vec<String>, rows: Vec<Vec<Value>> }` directly from the protobuf — `columns` is sourced verbatim from `proto.column_names` and each row is a positional value vector. The previous pipeline allocated one `BTreeMap<String, Value>` per row, cloned column names into the map keys (one clone per cell), then `rows_to_columnar` re-extracted the column list from the first row's keys and removed each cell into a `Vec<Vec<Value>>`. On a max-size 5000-row × 8-column result that's ~5k BTreeMap allocations + ~40k name clones + ~40k log-K removals — now zero of each.
- **One observable wire change**: column ordering on the JSON response flips from lexicographic (BTreeMap iteration order) to **SELECT-clause order** (the order of names in `proto.column_names`). This is a strict improvement (`SELECT b, a, c` now actually serializes columns in `["b", "a", "c"]` order) and the columnar shape itself was only introduced in this same release cycle, so no agent has accumulated a dependency on the lexicographic order. New `decode_preserves_proto_column_order` unit test (`src/query.rs`) and `e2e_smoke` regression assertion pin the property.
- **`outputSchema` + `structuredContent` for 10 tools (MCP 2025-06-18)**: every data-returning tool now returns `rmcp::Json<T>`, which auto-registers `outputSchema` on the `tools/list` entry and emits `structuredContent` on `CallToolResult`. Tabular tools (`execute_sql`, `list_processes`, `list_threads_in_process`, all five `chrome_*`) return `Json<DecodedTable>`; `list_tables` returns `Json<TableList>`; `list_table_structure` returns `Json<TableInfo>` (with a typed `ColumnInfo` per column). `load_trace` (text confirmation) and `list_stdlib_modules` (a JSON-encoded `String`, candidate for a future `Json<...>` migration) keep their existing return types.
- **Purely additive on the wire.** rmcp 0.16's `CallToolResult::structured` populates both `structured_content` *and* `content[0].text` (the latter via `Value::to_string()`, byte-equivalent to the previous manual `serde_json::to_string`), so older MCP clients that don't yet read `structuredContent` keep working with no adaptation.
- **Removed**: the `format` parameter on `execute_sql` and the `SqlResultFormat` enum (introduced earlier in the same cycle, unused by any released agent). `rows_to_columnar` helper deleted. Tool descriptions trimmed of redundant shape spell-outs (`outputSchema` is the formal contract now) — saves ~230 tokens per `tools/list` response (measured byte delta).
- **`list_tables` / `list_table_structure` surface SQLite contract violations as errors.** Pre-v0.9.0 the schema-discovery handlers swallowed upstream contract drift: `list_tables`'s `filter_map` silently dropped any non-string row from `sqlite_master`, and `list_table_structure`'s projection used `unwrap_or("?")` for `name` / `type`, producing `ColumnInfo { name: "?", data_type: "?", nullable: true }` placeholders that looked like real columns an agent could try to query. Now that `outputSchema` formalizes the wire shape, both paths surface the violation as the tool's `Err` return. The PRAGMA `notnull` lookup keeps `unwrap_or(0)` (= `nullable: true`) because exotic vtables can legitimately produce NULL there. Extracts a `pragma_row_to_column_info` helper used by both production and the regression tests, eliminating the prior test/production drift risk.
- **Tests**: 7 new `query.rs` unit tests (cell lookup, serialize shape, column-order preservation, empty-table envelope), 9 new `server.rs` unit tests (description regressions, `outputSchema` presence, schema-shape consistency across tabular tools, `TableList` / `TableInfo` / `ColumnInfo` shape pins, PRAGMA contract-violation surfacing for missing `name` / `type`), 1 new `e2e_smoke` assertion pinning the SELECT-order property end-to-end.

### [0.8.8](https://github.com/0xZOne/perfetto-mcp-rs/releases/tag/v0.8.8) (April 28, 2026)

- **Hotfix on top of v0.8.7's CJK-path support.** v0.8.7 correctly delivered the Chinese path through `trace_processor_shell.exe`'s argv (verified end-to-end: trace loads, HTTP server binds, /status responds), but `load_trace` still timed out at 20s on real CJK-locale Windows hosts. Root cause was one layer above the resolver: `trace_processor_shell` echoes the argv path bytes **raw** into `/status`'s `loaded_trace_name` protobuf field — on a cp936 host that means cp936-encoded bytes (`低端机` → `\xb5\xcd\xb6\xcb\xbb\xfa`) inside what was declared `optional string`. prost rejects those bytes as invalid UTF-8 and the entire `StatusResult::decode()` returns `Err`, so `wait_ready`'s `check_status().await.is_ok()` was permanently false even though stderr fired both Ready markers. The 20s deadline elapsed and `load_trace` reported "did not become ready" with no upstream signal as to why.
- Two-line wire fix: `loaded_trace_name` retyped from `optional string` to `optional bytes` in `proto/trace_processor.proto`. Wire format is identical (both length-delimited), so the change is forward- and backward-compatible with every `trace_processor_shell` build. Consumers (`tp_manager::loaded_name_matches`, `server::format_loaded_trace_display`) lossy-decode the bytes via `String::from_utf8_lossy` for display and matching — replacement chars on CJK directory components are fine because the basename match always wins for our use case.
- `loaded_name_matches` gains a path-suffix fallback: if the full-path comparison fails, accept any loaded name that ends with `/<expected_basename>`. This rescues CJK paths whose directory portion came back mojibake'd from the lossy decode but whose ASCII basename (`round13_2_trace.bin`) survived intact. Existing exact-match and bare-basename matchers are unchanged; suffix-collision rejection (`foo.trace.1` ≠ `foo.trace`) still passes because `/foo.trace.1` does not end with `/foo.trace`.
- 2 new tests: `format_loaded_trace_display_matches_when_cjk_dir_arrives_as_cp936` (`src/server.rs`) constructs the exact byte stream a cp936 host's `/status` returns (literal `\xB5\xCD\xB6\xCB\xBB\xFA` directory bytes + ASCII basename + ` (28 MB)` suffix) and asserts the basename match returns the original Path verbatim. `status_result_decodes_when_loaded_trace_name_has_invalid_utf8` (`src/tp_manager.rs`) hand-builds a real `/status` protobuf wire payload with cp936 bytes inside field 1 and asserts `StatusResult::decode()` succeeds — locks the schema so a future revert to `optional string` would fail CI. Fix verified end-to-end on a real cp936 Win11 host.

### [0.8.7](https://github.com/0xZOne/perfetto-mcp-rs/releases/tag/v0.8.7) (April 28, 2026)

- **Fixes a v0.8.6 regression for CJK-locale users whose trace path's 8.3 alias is itself non-ASCII.** v0.8.6 added an 8.3-short-name path-rewrite gated on `is_ascii()` for both the canonical path *and* the resulting short name. NTFS 8.3 short-name generation runs under the system OEM codepage at directory creation, so on a cp936 (Chinese) Windows host a `低端机traces` directory keeps `低端机~1` as its short alias — still non-ASCII. The double `is_ascii()` check rejected that and surfaced the v0.8.6 fallback error, telling those users to copy to an ASCII path even though their system would have read the original path fine. Empirically confirmed on a cp936 host: `cmd /c trace_processor_shell.exe "C:\Users\…\低端机traces\…\round13_2_trace.bin"` loads the trace and enters the REPL — CreateProcessW UTF-16 → CP_ACP `char*` is lossless when every char in the path is in the active codepage.
- v0.8.7 replaces both `is_ascii()` checks with a `WideCharToMultiByte(CP_ACP, WC_NO_BEST_FIT_CHARS, …, &used_default)` round-trip probe (Vista+; well within MSRV). True iff every UTF-16 code unit has an exact CP_ACP mapping with no silent best-fit substitution. Resolution order: strip the `\\?\` verbatim prefix that `Path::canonicalize` adds (trace_processor_shell doesn't accept it) → if canonical round-trips, use it directly (cp936 + Chinese path; cp932 + Japanese path; ASCII anywhere) → else try `GetShortPathNameW` (cp1252 + Chinese path created on a cp1252 host) → else surface the actionable error. Cache-key invariant from v0.8.6 preserved: `get_client` keys instances on the canonical path, not the rewritten one, so two requests for the same trace via different aliases share one `trace_processor_shell` process. Net effect: cp936/cp932 users no longer hit the fallback at all; cp1252 users keep the short-name rescue when 8.3 is enabled.
- The two `extern "system"` blocks (`WideCharToMultiByte`, `GetShortPathNameW`) and their constants (`CP_ACP = 0`, `WC_NO_BEST_FIT_CHARS = 0x400`) are consolidated into one module-level `mod win32 { … }` instead of being declared per-function. Still no new crate dependencies — the v0.8.6 "no `windows-sys` for one extern" stance carries forward.
- 5 unit tests in `src/tp_manager.rs` (`#[cfg(windows)]`): `windows_cp_acp_lossless` accepts ASCII + empty, rejects supplementary-plane chars (U+1F600 — unrepresentable on every Windows ANSI codepage); `strip_verbatim_prefix` handles `\\?\C:\…`, `\\?\UNC\server\share\…`, and pass-through; the resolver returns a CP_ACP-clean path with verbatim prefix stripped (or surfaces an actionable error when neither canonical nor short name round-trips); the resolver rejects unrepresentable non-existent paths. The v0.8.6 ASCII-only-Unix test stays unchanged.

### [0.8.6](https://github.com/0xZOne/perfetto-mcp-rs/releases/tag/v0.8.6) (April 28, 2026)

- **Windows non-ASCII paths now load transparently**. `ensure_trace_path_supported` is replaced with `resolve_trace_path_for_shell` which, on Windows + non-ASCII canonical path, calls `GetShortPathNameW` (kernel32.dll, `extern "system"` FFI — no new crate dependency) and substitutes the NTFS-native 8.3 short alias (`低端机traces` → `E4BDB~1`) for the argv passed to `trace_processor_shell.exe`. The cache key remains the canonical path, so two requests for the same trace under different aliases share one instance. Users no longer need to manually copy traces to `C:\traces\`. Internal `refactor: simplify v0.8.5 output drains` (`744837a`) is folded into this release.
- Fallback chain when 8.3 generation is disabled on the volume (`fsutil 8dot3name query`) or any path component lacks a short alias: returns the v0.8.5 LLM-friendly error, now with three workarounds — copy to ASCII path, re-enable 8.3 names, or pass an existing short name from `dir /x`.
- 4 new tests in `src/tp_manager.rs`: ASCII pass-through, Unix non-ASCII pass-through (argv is byte-exact on Unix, no rewrite needed), `#[cfg(windows)]` integration test that creates a real `低端机traces/` directory + file in a `tempdir` and asserts the rewritten path is ASCII (skips gracefully if 8.3 generation is disabled on the temp volume), and `#[cfg(windows)]` rejection test for non-existent non-ASCII paths. The Windows tests run automatically in the windows-amd64 release.yml job.
- Upstream: this is fundamentally a `trace_processor_shell.exe` bug — its `int main(int, char*[])` entry point loses non-ANSI-codepage characters. Chromium's `base::CommandLine::Init` solves it via `GetCommandLineW` + `CommandLineToArgvW`. An issue will be filed at https://github.com/google/perfetto/issues; until upstream fixes the entry point, this client-side rewrite is the lowest-friction path.

### [0.8.5](https://github.com/0xZOne/perfetto-mcp-rs/releases/tag/v0.8.5) (April 28, 2026)

- **Diagnostics fix**: `load_trace` now surfaces actionable error signals when `trace_processor_shell` fails to start, instead of bottoming out at the generic 20s readiness timeout. Three independent leaks plugged in `src/tp_manager.rs`:
  - **stdout was discarded** (`Stdio::null()` at `spawn`). Some Perfetto failure paths print to stdout; `Stdio::piped()` plus a new `spawn_stdout_drain` (folds into the same shared tail with a `[stdout] ` prefix) preserves them. The `format_stderr_tail` label is updated to `output tail (stderr + stdout, last N lines)` so the LLM sees both streams are folded in.
  - **stderr drain died on the first non-UTF-8 byte**. Tokio's `BufReader::lines()` returns `InvalidData` on any byte sequence that isn't valid UTF-8 and our drain `break`'d on Err — so on a Chinese-locale Windows shell, where trace_processor_shell prints error text via the active ANSI codepage (GBK / Shift-JIS / etc.), the very first error line killed the drain and every subsequent stderr byte was lost. Replaced with a manual `read_until(b'\n')` plus `String::from_utf8_lossy`: invalid sequences become `U+FFFD` (`�`) but the drain stays alive, ASCII portions (errno codes, port numbers, file sizes, paths under cp1252-overlap) survive intact, and startup needle matching keeps working because the needles are pure ASCII.
  - **Non-ASCII trace paths failed silently as a 20s timeout on Windows**. `trace_processor_shell.exe` consumes argv via `int main(int, char*[])` — bytes are decoded against the active ANSI codepage, so any character outside that codepage (Chinese paths under non-CJK locales, Japanese names under English locales, etc.) becomes `?`, fopen fails on a path that does not exist, and the shell exits with no actionable banner. New `ensure_trace_path_supported` runs after `canonicalize` and bails before spawn with a directly-actionable error: it reproduces the offending path and points at the two known workarounds — copy/move to an ASCII-only path, or pass the 8.3 short name from `cmd /c dir /x "<parent>"`. Unix is a no-op (argv is byte-exact).
- 5 new unit tests in `src/tp_manager.rs`: lossy decode preserves ASCII prefix and replaces invalid bytes with `U+FFFD`, LF / CRLF / bare-CR handling, `format_stderr_tail` label coverage, ASCII path always accepted, Unix non-ASCII no-op, and `#[cfg(windows)]` non-ASCII rejection (runs only in the Windows CI matrix).
- **Why this still warrants the v0.8.4 20s timeout**: the two changes target disjoint failure modes. v0.8.4 raised the deadline so a healthy-but-slow startup (low-end Windows + DirectWrite cold-cache font enumeration during stdlib init) doesn't time out; v0.8.5 makes shell-crash failures fail fast with rich diagnostics. The poll loop's `try_wait()` already detects an exited child within one `READY_POLL_INTERVAL`, so crash-mode failures resolve in ~100–300ms — they never approach the 20s ceiling.

### [0.8.4](https://github.com/0xZOne/perfetto-mcp-rs/releases/tag/v0.8.4) (April 27, 2026)

- **UX fix**: `--startup-timeout-ms` default raised from `5000` to `20000` (`src/tp_manager.rs:35`, `src/main.rs:90`). Surfaced by a real Quark-on-low-end-Windows session loading a ~11s Chrome trace where `trace_processor_shell` stdlib initialization plus `DWriteFontProxy` cold-cache font enumeration consistently pushed readiness past 5s. The 5s deadline was a hard fail — `wait_ready` (`src/tp_manager.rs:115-223`) bails the entire `load_trace` call rather than surfacing a soft signal — so MCP clients saw the wrapper repeatedly restart `trace_processor_shell` and eventually try to bypass it by invoking the shell directly. Why a single bumped default instead of a tiered "soft 5s → hard 30s" retry: `wait_ready` is already a poll loop that returns the instant the stderr ready marker (or the `STATUS_FALLBACK_DELAY` /status fallback after 1s) fires. Healthy traces still resolve in ~1–2s and never approach the deadline; only the slow path benefits from the longer ceiling. A tiered retry inside the same poll loop is observationally equivalent to extending the deadline. Override entrypoints (`PERFETTO_STARTUP_TIMEOUT_MS` env, `--startup-timeout-ms` flag) unchanged from v0.8.0; CI/scripted contexts that want fast-fail can still pin a smaller value. README tables (en + zh) updated to match.

### [0.8.3](https://github.com/0xZOne/perfetto-mcp-rs/releases/tag/v0.8.3) (April 20, 2026)

- **Agent-guidance fix**: the `chrome_*` preflight error on non-Chrome traces now points the agent to the recovery tool instead of dead-ending. When a trace lacks `chrome.process_type` track-descriptor args, the error message appended `Call list_stdlib_modules to discover modules that fit this trace, then query via execute_sql.` Prior wording only said "use execute_sql for a different trace type", and observation samples (v0.7 #2, v0.8 #1) consistently showed agents falling back to `list_processes` + hand-rolled 4-table JOINs (`slice JOIN thread_track JOIN thread JOIN process`) instead of the stdlib path. v0.8 sample #2 on the same Quark/sina trace — with this fix deployed — recorded 3/3 `execute_sql` calls using `INCLUDE PERFETTO MODULE` (`chrome.page_loads`, `chrome.tasks`), 0 raw `slice` scans. All five `chrome_*` handlers share the same `ensure_chrome_trace` preflight (`src/server.rs`), so the redirect applies uniformly. The integration test `all_chrome_handlers_reject_non_chrome_via_preflight` now asserts the `list_stdlib_modules` substring in every handler's error (previously asserted only in the first), so future refactors can't silently delete the redirect for any single tool.

### [0.8.2](https://github.com/0xZOne/perfetto-mcp-rs/releases/tag/v0.8.2) (April 19, 2026)

- **Windows fix**: `uninstall` no longer fails hard when the cache directory can't be removed due to locked files (a running MCP client holding `trace_processor_shell.exe` open). `clean_cache` now classifies `ErrorKind::PermissionDenied` as Skipped-with-guidance ("close Claude Code / Codex and retry, or delete manually") instead of Failed. Claude/Codex deregistration — the meaningful work — already succeeded at that point, so the wrapper correctly proceeds to remove the binary. Cache leftovers are harmless (they just re-populate on the next install).
- **Windows fix**: `uninstall.ps1` no longer discards the binary's stdout. The v0.8.1 PR accidentally added `| Out-Null` around the `_invokeNative { & $dest uninstall ... }` call as part of an EAP-dance refactor, which ate the step-by-step progress output (`==> Claude: deregistered...`, `==> Codex: deregistered...`, etc.). Only stderr was visible, making users think only errors happened. Dropping `| Out-Null` restores the flow; `_invokeNative` still manages `$ErrorActionPreference` and `$LASTEXITCODE` correctly. Surfaced by a real Windows PowerShell uninstall run.

### [0.8.1](https://github.com/0xZOne/perfetto-mcp-rs/releases/tag/v0.8.1) (April 19, 2026)

- **Windows fix**: `install` subcommand now correctly invokes `codex` (and any other CLI) when the installed form is a `.cmd`/`.bat` shim instead of a native `.exe`. npm-installed CLIs like OpenAI's Codex land as `codex.cmd` on Windows; v0.8.0's `Command::new("codex")` failed with `program not found` because `CreateProcessW` can't exec batch files directly. v0.8.1 resolves the path through `which::which` (applies PATHEXT consistently) and on Windows wraps `.cmd`/`.bat` shims through `cmd /c`. ### [0.8.0](https://github.com/0xZOne/perfetto-mcp-rs/releases/tag/v0.8.0) (April 19, 2026)

- **New `install` / `uninstall` subcommands**. The binary now self-registers with Claude Code (`claude mcp add`) and Codex (`codex mcp add`) and self-cleans its `dirs::data_local_dir()` cache, replacing the register/deregister blocks that lived in `install.sh` / `install.ps1` / `uninstall.sh` / `uninstall.ps1` (four copies of the same CLI-schema + cache-path knowledge that had already drifted into bugs — e.g. the Git Bash `LOCALAPPDATA` cygpath issue fixed in v0.7.0's uninstall.sh). Default subcommand-less invocation still runs the MCP server, so existing client configs don't need to change. New flags: `install --scope user|local|project`, `install --binary-path <P>`, `uninstall --scope X [--keep-cache]`, and `--skip-claude` / `--skip-codex` on both.
- **Shell scripts shrink to distribution glue** — `install.sh` and `uninstall.sh` (plus PowerShell counterparts) now do download / platform detect / file-lock aside / PATH hint / binary delete, and delegate all Claude/Codex registration + cache cleanup to `perfetto-mcp-rs install|uninstall`. Uninstall scripts **preserve the binary on subcommand failure** so users can fix the underlying issue (locked cache, wrong project dir, broken config) and re-run — only a successful uninstall (or confirmed v0.7 "no subcommand" case, or already-missing binary) deletes the binary.
- **`SCOPE` env var** on all four scripts (default `user`) plumbs through to `--scope`. For local/project installs, run the script from the target project directory: `SCOPE=local bash install.sh` / `SCOPE=local bash uninstall.sh`. Claude stores local/project scope in a CWD-keyed way inside `~/.claude.json` / project's `.mcp.json`, so the CWD must match at both install and uninstall time.
- **No more `claude mcp list` probe on the install/uninstall paths**. We used to list before add/remove to distinguish first-install from upgrade, but `claude mcp list` is **not a passive probe**: it skips the workspace-trust dialog and spawns every visible stdio server to run a health check. Running that from `--scope local|project` in a hostile or broken project directory is unsafe (arbitrary command execution from that repo's `.mcp.json`) and unreliable (hangs / fails on any broken sibling server). v0.8.0 always calls `mcp remove` first and classifies its stderr: `No <scope>-scoped MCP server found with name: ...` (exit 1) is benign; anything else — including output that mixes corruption-recovery text with the benign line — is a hard failure. See `src/install.rs::claude_remove_error_is_not_found` + its unit tests for the exact pattern.
- **`PERFETTO_STARTUP_TIMEOUT_MS` / `PERFETTO_QUERY_TIMEOUT_MS` are read lazily inside `run_server`**, not by clap at top-level parse. A stale or invalid value in the user's shell no longer blocks `perfetto-mcp-rs install` / `uninstall` with a parse error — those maintenance commands don't care about server timeouts. Precedence preserved: explicit `--flag` > env var > built-in default; invalid env in server mode still fails loudly.
- `install.rs::run_install` absolutizes `--binary-path` via `std::path::absolute` before passing it to the MCP clients. MCP clients spawn the registered command from their own working directory, so a relative `./perfetto-mcp-rs` would silently break. We deliberately don't `canonicalize` (that would resolve symlinks and diverge from the `$INSTALL_DIR` path users see). Windows POSIX paths (`/c/Users/...` from Git Bash) are converted by the shell wrapper via `cygpath -m` before reaching the Rust binary.
- **Migration 0.7.x → 0.8.0**: re-run `install.sh` (or `install.ps1`) — the new binary brings the subcommands and subsequent `uninstall.sh` picks them up. Uninstalling a 0.7.x install with the new `uninstall.sh` only deletes the binary and prints a manual-cleanup hint, because the old binary has no `uninstall` subcommand. Manual cleanup command bundle for reference:
  ```sh
  claude mcp remove perfetto-mcp-rs --scope user
  codex  mcp remove perfetto-mcp-rs
  # Cache (pick the one for your OS):
  rm -rf "${XDG_DATA_HOME:-$HOME/.local/share}/perfetto-mcp-rs"   # Linux
  rm -rf "$HOME/Library/Application Support/perfetto-mcp-rs"      # macOS
  # Windows Git Bash/MSYS ($LOCALAPPDATA is a native 'C:\...' path that
  # POSIX rm can't walk; convert with cygpath, or use PowerShell):
  rm -rf "$(cygpath -u "$LOCALAPPDATA")/perfetto-mcp-rs"          # Git Bash
  # PowerShell: Remove-Item -Recurse -Force "$env:LOCALAPPDATA\perfetto-mcp-rs"
  ```
- Ship includes 23 hermetic integration contracts (`tests/install_subcommand.rs`) exercising the subcommands via stateful shell fixtures (`tests/fixtures/{claude,codex}`), plus 12 unit tests in `src/install.rs`. Integration tests run under a restricted `PATH=<fixtures>` so `which::which` can never find real CLIs. Design in `docs/plans/v0.8-binary-self-install.zh-CN.md`.
- `cargo install perfetto-mcp-rs` remains a **developer-fallback channel** only — it requires a system `protoc` because `build.rs` invokes `prost-build`. End users should use `install.sh` / `install.ps1`.

### [0.7.0](https://github.com/0xZOne/perfetto-mcp-rs/releases/tag/v0.7.0) (April 18, 2026)

- **Breaking**: tool surface change from v0.6. Four Chrome domain tools return as row-preserving thin wrappers over PerfettoSQL stdlib views, plus one new tool. v0.5→v0.7 upgraders see a fully new Chrome tool set; v0.6→v0.7 see these as additions.
- **Breaking (behavioral)**: `chrome_scroll_jank_summary` keeps its v0.5 name but **changes output structure** from pre-aggregated frequency (`COUNT(*) GROUP BY cause_of_jank`) to row-level worst-frame output (sorted by `delay_since_last_frame DESC`, limit 100). The name is preserved for consistency with the other `chrome_*_summary` tools — all five Chrome tools return row-level top-N overviews of their domain entity. New columns: `sub_cause_of_jank`, `event_latency_id`, `scroll_id`, `vsync_interval` — per-frame data lets agents group, filter, and correlate rather than stopping at a pre-baked table. v0.5.0 users must adapt to the new output shape even though the tool name is unchanged.
- `chrome_main_thread_hotspots` uses `thread.is_main_thread = 1` (Linux convention: `tid == pid`, populated in `trace_processor` when thread metadata is present) to identify main threads — more robust than filtering by thread name strings that could break across Chrome versions. Adds `task_type`, `cpu_pct` (thread_dur/dur), `thread_dur_ms`. **Caveat**: `is_main_thread` is `CppOptional` and may be NULL for traces lacking complete thread creation metadata; empty result means retry with `execute_sql WHERE thread_name IN ('CrBrowserMain', 'CrRendererMain')`.
- `chrome_page_load_summary` and `chrome_startup_summary` add `id` (and `browser_upid` for startup) for correlation queries. New `chrome_web_content_interactions` surfaces clicks, taps, and keyboard interactions ranked by duration for INP analysis.
- Design rationale: v0.3/v0.4/v0.6 observation samples across Claude and Codex show `tools/list` channel drives stable behavior — tools get called reliably. The description-URL / `instructions` channel shows high variance: most sessions had 0/N INCLUDE, one had ≥4 INCLUDE but with two wrong module names (`chrome.chrome_scrolls` and `chrome.chrome_loads` don't exist). Both failure modes — not using stdlib, and using it with wrong names — are solved by domain tools (skip the name) + `list_stdlib_modules` (provide the correct name). One session demonstrated agents depth-drilling to call-stack depth 7 (DWriteFontProxy sync IPC, forced synchronous layout) — only possible with row-level output, validating the row-preserving design. v0.6 removed Chrome domain tools based on the "answer-shaped" critique of v0.5's pre-aggregated SQL — but that critique applied to implementation, not direction. v0.7 restores the tools with row-preserving SQL following the Perfetto stdlib's own "high-level domain tables" design intent.
- Add `list_stdlib_modules` tool: returns a curated JSON array of PerfettoSQL stdlib modules (module name, domain, views, description, illustrative usage). Takes no parameters — call before loading a trace to discover which stdlib module covers analyses **not** handled by the dedicated `chrome_*` tools (Android tools, `slices.with_context`, etc.). Positioned as auxiliary discovery, not the primary path.

### [0.6.0](https://github.com/0xZOne/perfetto-mcp-rs/releases/tag/v0.6.0) (April 17, 2026)

- **Breaking**: remove the four pre-built Chrome domain tools shipped in v0.5.0 (`chrome_scroll_jank_summary`, `chrome_page_load_summary`, `chrome_main_thread_hotspots`, `chrome_startup_summary`) along with their SQL constants, the shared `ChromeTraceParams` struct, and the `format_chrome_tool_error` helper. The tools were answer-shaped wrappers whose fixed SQL and narrow column sets risked steering even top-tier LLMs away from the richer analyses they could do with their own SQL against the stdlib. `chrome_main_thread_hotspots` additionally dropped `ipc_hash` / `posted_from` attribution columns and did not actually filter to the main thread. Clients invoking these tool names will fail; re-express via `execute_sql` with an appropriate stdlib module (e.g. `chrome.scroll_jank.scroll_jank_v3`, `chrome.page_loads`, `chrome.tasks`, `chrome.startups`).
- Augment the `execute_sql` tool description with stdlib guidance following the Google `com.google.PerfettoMcp` plugin's single-tool pattern — subtopic URL list (stdlib-docs with anchor-based per-package fetch guidance, perfetto-sql-syntax, frametimeline, cpu-scheduling, memory-counters, battery-counters, android-log) and Google's "worth fetching fully" phrasing. The description carries the nudge on the `tools/list` channel v0.4 confirmed reaches agents. The v0.4-era "`INCLUDE PERFETTO MODULE` must be in a separate execute_sql call" warning is removed — empirical testing against `trace_processor_shell` v54.0 (recorded in `docs/google-perfetto-mcp-analysis.md`) shows the combined `INCLUDE X; SELECT FROM X_view` form works, and the prior caveat was actively steering agents toward a two-call pattern they do not need. The description also carries a single stdlib-first signal only — no raw-table fallback note (prelude is part of the stdlib and `list_tables` already tells the agent which tables are directly available).
- Server `instructions` retain the v0.4 curated stdlib module directory. v0.3/0.4 samples showed the `instructions` channel is not behavior-active for the current Claude Code agent, but the handshake token cost is already paid and the directory is a free hedge for future agent frameworks or MCP clients that do route `instructions` into their system prompt. Multi-channel redundancy is by design.
- New `tests/e2e_stdlib_include.rs` with two cases that replace the behavior coverage removed alongside the Chrome domain tools. The first exercises a single-call `INCLUDE PERFETTO MODULE slices.with_context; SELECT ... FROM thread_slice` against `basic.perfetto-trace` — the generic regression net for the description's "stdlib-first, single-call" posture against upstream `trace_processor_shell` changes. The second exercises exactly the `chrome.scroll_jank.scroll_jank_v3` → `chrome_janky_frames` SQL printed in `README.md` as the canonical replacement for the removed `chrome_scroll_jank_summary` tool, run against `tests/fixtures/scroll_jank.pftrace` (kept for this reason; the original Chrome fixture removal would have left the README example unverified). `tests/fixtures/page_loads.pftrace` is still removed — no README example depends on it.
